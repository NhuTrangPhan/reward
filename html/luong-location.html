Luồng lấy vị trí (Location)

Zalo Mini App "vibe" — ví dụ trang 72 (Feature QR Garage) · cơ chế dùng chung mọi vibe app · xác minh theo code 2026-06-19

  1. Zalo không trả lat/lng thẳng — getLocation() chỉ trả một token; phải decode ở backend kèm secret_key riêng từng Mini App.
  2. Vibe không tự gọi native được — 2 bản zmp-sdk trong 1 webview tranh biến global window.onNativeMessage → lỗi -1401. Vibe nhờ host gọi giùm qua CustomEvent.
  3. Luồng 2 giai đoạn: (1) trong webview lấy token; (2) backend đổi token → lat/lng.
  4. Secret lấy ở mini-app-backend; decode chạy ở Frontend API. cnv-id KHÔNG tham gia luồng này.
trong máy (webview Zalo) backend CNV phía Zalo (bên ngoài) điểm mấu chốt — nét liền = gửi đi · - - nét đứt = trả về

Sơ đồ luồng tổng (gộp 2 giai đoạn)

Một mạch liền: token + accessToken lấy ở giai đoạn 1 được mang nguyên sang giai đoạn 2 (bước ⑥) để backend decode. Đường kẻ dọc là ranh giới trong máy ↔ trên server.

Phóng to từng chặng

Giai đoạn 1 — Lấy "token vị trí" (trong webview)

Vì sao phải "giành" window.onNativeMessage?

zmp-sdk nhận kết quả native bằng cách Zalo gọi hàm global window.onNativeMessage(serialId, data). Mỗi bản SDK giữ một bảng callbacks[serialId] riêng nhưng đăng ký vào cùng một ô global duy nhất. Vibe nạp sau ghi đè ô này → nếu host gọi native mà ô đang trỏ bảng của vibe thì callback rơi mất → treo. Nên host phải giành lại ô về bản mình ngay trước khi gọi, rồi trả lại sau.

Giai đoạn 2 — Đổi token ra toạ độ thật (backend)

Secret chỉ đi nội bộ giữa các service — KHÔNG ra client. Decode (gọi Zalo Graph) chạy NGAY tại Frontend API. cnv-id KHÔNG tham gia luồng location.

Chi tiết HTTP

vibe → Frontend API

POST {environment.frontendApiUrl}/location/resolve
Headers:
  X-Mini-App-Id:    <config.miniAppId>        // BẮT BUỘC
  X-Customer-Token: <auth.customerToken()>    // BẮT BUỘC (phải đã đăng nhập)
Body:
  { "code": "<token vị trí>", "accessToken": "<access token>", "accountDomainName": "<...>" }

Backend lấy accountId từ user đã xác thực (qua X-Customer-Token), KHÔNG từ accountDomainName trong body.

Frontend API → mini-app-backend (lấy secret)

POST {integration.cnv-internal.url}/info/get-mini-app-secret
Body: { "accountId": <id>, "miniAppId": "<id>" }
→ { "data": { "appId": "...", "appSecret": "..." } }   // lệch account → rỗng

Port 8092, network nội bộ, KHÔNG ký HMAC (bảo mật dựa vào port chặn internet + kiểm sở hữu account).

Frontend API → Zalo Graph (decode)

GET {integration.zalo.graph.url|https://graph.zalo.me/v2.0}/me/info
Headers: access_token, code, secret_key
→ data.latitude / data.longitude

Những điểm dễ hiểu sai

  1. getLocation() trả token, KHÔNG phải lat/lng.
  2. Vibe đừng tự gọi getLocation/getAccessToken → -1401. Luôn qua host bằng vibe-get-location.
  3. secret_key lấy ở mini-app-backend, không phải cnv-id.
  4. Decode chạy ngay tại Frontend API, không relay đi đâu.
  5. /location/resolve bắt buộc đã đăng nhập.
  6. Header gọi Zalo Graph là access_token, code, secret_key (đặt ở HTTP header).
📜 Lịch sử — phương án đã bỏ (relay qua cnv-id)

Bản đầu định relay qua cnv-id (thêm ResolveLocation + GetLocationByAccessToken ở cnv-id, IdInternalService.resolveLocation() ở Frontend API gọi sang cnv-id /internal/zalo/location/resolve). Đã thay bằng cách hiện tại (Frontend API tự xin secret ở mini-app-backend + tự decode). Grep repo cnv-id hiện tại không còn các symbol này.

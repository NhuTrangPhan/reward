OAuth 2.0 — Tham khảo trực quan

Luồng Authorization Code Grant + PKCE, các chuẩn RFC chính và những nền tảng lớn đã triển khai mô hình Connected Accounts tương đương.

1

Abstract Protocol Flow (RFC 6749 §1.2)

Sơ đồ gốc trong chuẩn RFC 6749 mô tả tương tác trừu tượng giữa bốn vai trò OAuth — Client, Resource Owner, Authorization Server, Resource Server — qua sáu bước (A → F). Bên trái là ASCII nguyên văn từ chuẩn; bên phải là cùng sơ đồ, nhưng các vai trò được ánh xạ vào bối cảnh CNV CDP × GotIt.

RFC 6749 · FIGURE 1 — ABSTRACT PROTOCOL FLOW
     +--------+                               +---------------+
     |        |--(A)- Authorization Request ->|   Resource    |
     |        |                               |     Owner     |
     |        |<-(B)-- Authorization Grant ---|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(C)-- Authorization Grant -->| Authorization |
     | Client |                               |     Server    |
     |        |<-(D)----- Access Token -------|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(E)----- Access Token ------>|    Resource   |
     |        |                               |     Server    |
     |        |<-(F)--- Protected Resource ---|               |
     +--------+                               +---------------+

                  Figure 1: Abstract Protocol Flow
BIẾN THỂ · CNV CDP × GOTIT
AAuthorization Request — CNV CDP gửi yêu cầu xin quyền. Theo khuyến nghị RFC 6749, bước này nên đi gián tiếp qua Authorization Server (GotIt OAuth) thay vì trực tiếp tới doanh nghiệp — tức là CNV CDP redirect trình duyệt doanh nghiệp đến /oauth/authorize.
BAuthorization Grant — Doanh nghiệp (Resource Owner) cấp quyền. Trong Authorization Code Grant (RFC 6749 §4.1), "grant" này là một authorization_code vòng đời ngắn, dùng một lần.
CAuthorization Grant → Auth Server — CNV CDP gọi server-to-server POST /oauth/token, đính kèm authorization_code, client_id/client_secret và code_verifier (PKCE · RFC 7636).
DAccess Token — GotIt OAuth Server xác thực Client, kiểm tra grant, nếu hợp lệ phát access_token (Bearer · RFC 6750) và refresh_token (RFC 6749 §6).
EAccess Token → Resource Server — CNV CDP gọi GotIt Biz API v5.0 kèm header Authorization: Bearer {access_token}. Resource Server xác định Resource Owner qua token.
FProtected Resource — GotIt Biz API trả tài nguyên được bảo vệ (voucher code, trạng thái ví, campaign…) và billing vào đúng ví của doanh nghiệp đã cấp quyền.

3

Kiến trúc thành phần và tương tác chi tiết

Sơ đồ mô tả các thành phần UI và luồng dữ liệu thực tế khi doanh nghiệp cấp quyền cho CNV CDP truy cập tài khoản GotIt của mình. Luồng chính (1–8) diễn ra một lần tại thời điểm liên kết; luồng Web API (A–D) lặp lại mỗi khi CNV CDP phát hành voucher.

1Doanh nghiệp truy cập CNV CDP — đăng nhập nền tảng CNV CDP và chọn cấu hình "Ví GotIt của doanh nghiệp" làm nguồn phát hành voucher.
2Redirect tới /oauth/authorize — CNV CDP điều hướng trình duyệt sang GotIt kèm client_id, scope, redirect_uri và code_challenge (PKCE · RFC 7636).
3Đăng nhập và chấp thuận — Doanh nghiệp xác thực trên miền GotIt, xem danh sách quyền CNV CDP yêu cầu và bấm "Chấp thuận". CNV CDP không bao giờ thấy mật khẩu GotIt.
4GotIt sinh authorization_code — token dùng một lần, vòng đời ngắn (≤ 10 phút), chỉ dùng để đổi lấy access_token ở bước kế tiếp.
5Redirect về CNV CDP kèm ?code=… — GotIt trả trình duyệt về redirect_uri mà CNV CDP đã đăng ký.
6Đổi code lấy token — CNV CDP gọi trực tiếp (server-to-server) POST /oauth/token với code và code_verifier, nhận về access_token + refresh_token.
7Lưu token gắn với doanh nghiệp — CNV CDP lưu access_token trong token store, liên kết với mã doanh nghiệp để dùng cho các request phát hành voucher sau này.
8Gọi Biz API v5.0 — mỗi lần phát hành voucher, CNV CDP gửi Authorization: Bearer {access_token} (RFC 6750). GotIt xác định Resource Owner từ token → billing trừ đúng ví doanh nghiệp.
AIntrospection request (tuỳ chọn) — nếu GotIt phát opaque token, Biz API có thể gọi POST /oauth/introspect (RFC 7662) để kiểm tra token còn hiệu lực.
BIntrospection response — Authorization Server trả metadata active, sub (biz_id), scope, exp. Nếu dùng JWT self-contained thì bỏ qua A–B.
CTrừ ví theo sub — Biz API dùng sub trong token để định tuyến billing vào đúng ví doanh nghiệp đã chấp thuận.
DPhản hồi CNV CDP — Biz API trả voucher code / trạng thái phát hành. CNV CDP gửi kết quả cho doanh nghiệp trong dashboard.

RFC 6749

The OAuth 2.0 Authorization Framework

Chuẩn gốc — định nghĩa toàn bộ khung OAuth 2.0 mà mọi implementation phải tuân thủ.

  • 4 role: Resource Owner · Client · Authorization Server · Resource Server
  • 4 grant type: Authorization Code · Implicit · Password · Client Credentials
  • Endpoint /authorize và /token, error response chuẩn
  • Token lifetime, refresh_token, scope
Đọc RFC 6749 →
RFC 6750

Bearer Token Usage

Cách Client gửi access_token và Resource Server xác minh nó.

  • Header Authorization: Bearer {token}
  • Response WWW-Authenticate khi token sai/hết hạn
  • Bắt buộc TLS cho toàn bộ traffic
  • Error codes: invalid_token, insufficient_scope
Đọc RFC 6750 →
RFC 7009

OAuth 2.0 Token Revocation

Client hoặc Resource Owner chủ động vô hiệu hoá token trước khi hết hạn.

  • Endpoint /oauth/revoke
  • Hỗ trợ cả access_token và refresh_token
  • Hint token_type_hint giúp server tìm nhanh
  • Khuyến nghị phát webhook oauth.revoked cho Client
Đọc RFC 7009 →
RFC 7636

Proof Key for Code Exchange (PKCE)

Bảo vệ Authorization Code Grant khỏi tấn công intercept code tại mobile/SPA/public client.

  • code_verifier: chuỗi 43–128 ký tự URL-safe
  • code_challenge = BASE64URL(SHA256(code_verifier))
  • Method S256 được khuyến nghị (plain đã bị deprecated)
  • Bắt buộc cho public client, khuyến nghị cho mọi client
Đọc RFC 7636 →
BCP

OAuth 2.0 Security Best Current Practice

Bản cập nhật bảo mật mới nhất của IETF OAuth Working Group — tập hợp khuyến nghị rút ra từ 10+ năm vận hành thực tế.

  • PKCE bắt buộc cho mọi client (không chỉ public)
  • Không dùng Implicit Grant — chỉ dùng Authorization Code
  • Rotate refresh_token mỗi lần refresh
  • Strict validation redirect_uri (exact match, không wildcard)

Nền tảng tham khảo đã triển khai

Ba nền tảng lớn đã vận hành thành công mô hình OAuth Connected Accounts song song với API Key, không buộc đối tác hiện hữu phải migration.

Stripe Connect

Platform Payment Aggregator

Mô hình platform-mediated điển hình — Stripe cho phép cả API Key trực tiếp lẫn Connect OAuth cùng tồn tại hơn 10 năm.

  • API Key riêng cho merchant tự tích hợp trực tiếp
  • Connect OAuth cho platform gọi API thay mặt merchant
  • Phân biệt billing qua Stripe-Account header trong token
  • Merchant tự revoke quyền qua Stripe Dashboard

App Marketplace cho SaaS Partner

  • Partner OAuth cho SaaS app trên Shopify App Store
  • Scope granularity theo từng resource (orders, customers, products…)
  • Webhook app/uninstalled khi merchant gỡ app

Google OAuth 2.0

Reference Implementation

Implementation chuẩn RFC 6749/7636 được hàng triệu ứng dụng tin dùng — thường là nguồn tham khảo khi học OAuth đúng cách.

  • PKCE được yêu cầu cho tất cả client type
  • Incremental consent — xin thêm scope khi cần
  • Token revocation qua Google Account settings
  • Rotating refresh_token theo Security BCP
